INDEXING STRATEGY
-----------------------------------------------------------------------------
While MongoDB automatically indexes the `_id` field upon generic insertion, the Yelp dataset relies heavily upon custom string IDs (`business_id` and `user_id`). Custom indexes must be strictly defined to optimize the specific aggregation pipelines defined in Task 3. Without these, MongoDB would perform full collection scans (O(N) time complexity), resulting in Cartesian products during `$lookup` stages that cause queries to freeze on datasets exceeding 100,000 documents.

The following six indexes were engineered to minimize read times for the required analytical queries (slashing complex >15 minute Cartesian scans down to ~13 seconds) while carefully balancing the storage penalty associated with maintaining index overhead.

1. FOREIGN KEY INDEX: Reviews to Businesses
   - Collection: `reviews`
   - Command: db.reviews.createIndex({ "business_id": 1 })
   - Justification (Query 3.4): Tracing review behavior grouped by business category. When the database aggregates data, indexing `business_id` inside the `reviews` collection ensures retrieving all reviews for a specific business executes as an O(log N) B-Tree traversal rather than a linear scan of hundreds of thousands of reviews.

2. FOREIGN KEY INDEX: Reviews to Users
   - Collection: `reviews`
   - Command: db.reviews.createIndex({ "user_id": 1 })
   - Justification: This index acts defensively. Should a pipeline (like modifications to 3.5 or 3.6) trace from `users` -> `reviews`, it provides the necessary bridge index to pull all documents authored by a specific user instantly.

3. PRIMARY KEY INDEX: Businesses
   - Collection: `businesses`
   - Command: db.businesses.createIndex({ "business_id": 1 })
   - Justification (Query 3.2): Because `business_id` was automatically imported as a string field alongside the native `_id` mechanism, MongoDB does not automatically index it. Query 3.2 executes a `$lookup` scanning *from* `reviews` *to* `businesses`. Without this explicit index on the target `businesses` collection, the pipeline was forced to conduct a "Full Table Scan" of all 11,247 Indiana businesses for every single one of the 500,000 reviews. This index turns that O(R * B) operation into O(R * log(B)).

4. PRIMARY KEY INDEX: Users
   - Collection: `users`
   - Command: db.users.createIndex({ "user_id": 1 })
   - Justification (Query 3.6): Identical logic to the business index bottleneck. Query 3.6 calculates elite status behaviors iteratively by doing a reverse `$lookup` originating from `reviews` directly back to `users`. This index guarantees the engine can instantly pinpoint the user authoring the review in O(1) matching time.

5. MULTIKEY INDEX: Business Categories
   - Collection: `businesses`
   - Command: db.businesses.createIndex({ "categories": 1 })
   - Justification (Queries 3.1, 3.2, 3.4, 3.7): Almost half the analytical requirements demand segmentation by category. Because `categories` is an embedded array, MongoDB creates a highly specialized Multikey Index, generating an index branch for *every* string inside the array. This empowers the grouping stages to instantly locate all businesses tagged with "Mexican" or "Nightlife" simultaneously without doing costly $unwind scans first.

6. COMPOUND SORT/COVERED INDEX: City, Volume, and Stars
   - Collection: `businesses`
   - Command: db.businesses.createIndex({ "city": 1, "review_count": -1, "stars": -1 })
   - Justification (Query 3.1): Task 3.1 explicitly demands clustering safest/least-safe cities sorted by star ratings and weighted by review volume. This compound index mirrors the aggregation pipeline's exact sorting directives. By grouping `city` first, and placing the other keys descending (-1), MongoDB can stream the answers sequentially directly out of the B-Tree lodged in RAM, culminating in an execution block of just 0.01 seconds.
