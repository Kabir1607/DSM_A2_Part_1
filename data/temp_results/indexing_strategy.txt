-----------------------------------------------------------------------------
INDEXING STRATEGY
-----------------------------------------------------------------------------
While MongoDB automatically indexes the `_id` field, custom indexes are required 
to optimize the specific aggregation pipelines defined in Task 3. Without these, 
MongoDB would perform full collection scans (O(N) time complexity), which is 
highly inefficient for datasets exceeding 100,000 documents.

The following four indexes were chosen to minimize read times for the required 
analytical queries, while carefully balancing the write-penalty associated with 
maintaining too many indexes.

1. FOREIGN KEY INDEX: Reviews to Businesses
   - Collection: `reviews`
   - Command: db.reviews.createIndex({ "business_id": 1 })
   - Justification: Task 3.4 requires analyzing review behavior (length, useful votes) 
     grouped by business category. Because categories live in the `businesses` collection 
     and the behavior metrics live in the `reviews` collection, a $lookup is mandatory. 
     Indexing the `business_id` inside the reviews collection ensures that when the database 
     joins the two, it performs an O(log N) index traverse rather than a full table scan 
     of all 1.5M+ reviews.

2. FOREIGN KEY INDEX: Reviews to Users
   - Collection: `reviews`
   - Command: db.reviews.createIndex({ "user_id": 1 })
   - Justification: Tasks 3.5 and 3.6 require comparing review behavior against user 
     metadata (tenure and elite status). Similar to Index #1, when aggregating from 
     `users` and performing a $lookup into `reviews` to fetch the text and ratings, 
     an index on `user_id` inside the reviews collection acts as the critical bridge, 
     slashing query execution time from minutes to milliseconds.

3. MULTIKEY INDEX: Business Categories
   - Collection: `businesses`
   - Command: db.businesses.createIndex({ "categories": 1 })
   - Justification: Tasks 3.1, 3.2, and 3.4 all require filtering, grouping, or 
     segmenting data by category. Because `categories` is an array, MongoDB will 
     create a Multikey Index, generating an index entry for every single category 
     string inside the array. This allows the aggregation pipeline's $match and $group 
     stages to instantly locate all businesses tagged as "Restaurants" or "Nightlife" 
     without inspecting every document.

4. COMPOUND SORT/COVERED INDEX: City, Volume, and Stars
   - Collection: `businesses`
   - Command: db.businesses.createIndex({ "city": 1, "review_count": -1, "stars": -1 })
   - Justification: Task 3.1 explicitly asks to identify the safest/least-safe cities 
     based on star ratings while accounting for review volume. This compound index 
     perfectly mirrors the sorting requirements of that query. By placing `city` first, 
     the database groups the data efficiently. Placing `review_count` and `stars` in 
     descending order (-1) allows MongoDB to return the top results directly from the 
     index in RAM (a Covered Query), completely bypassing the need to fetch the actual 
     documents from the hard drive.
